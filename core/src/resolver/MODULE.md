# resolver

**Purpose**
Deterministic cross-reference resolver (TDD §5.5, Methods 1 & 2). Walks every `RequirementTree` in a corpus and turns the parser's raw reference strings into typed, status-tagged manifests covering three flavors: internal (same plan), cross-plan (another plan in the same MNO/release), and standards (3GPP/GSMA citations with release version). Serves FR-4 (cross-reference extraction with section + release).

**Public surface**
- `CrossReferenceResolver(trees: list[RequirementTree])` (resolver.py) — constructor indexes by `plan_id` and `req_id`; `resolve_all()` returns manifests for the whole corpus; `resolve_tree(tree)` resolves one
- `CrossReferenceManifest` — per-document output: `plan_id`, `mno`, `release`, three typed ref lists, and a `ManifestSummary`; `to_dict`, `save_json`
- `ResolvedInternalRef`, `ResolvedCrossPlanRef`, `ResolvedStandardsRef` — resolution results with `status: RefStatus`
- `RefStatus` enum — `resolved | broken | unresolved` (see Invariants for the distinction)
- `ManifestSummary` — per-type totals + resolved/broken/unresolved counts
- `resolve_cli.main` — CLI entrypoint

**Invariants**
- **LLM-free.** Method 3 (concept linking) is delegated to [taxonomy](../taxonomy/MODULE.md); this module stays deterministic and cheap to re-run.
- `RefStatus` has a precise three-way meaning:
  - `resolved` — target found in the corpus
  - `broken` — target *ID/section* is a valid shape but missing from the corpus (likely a typo or a removed requirement)
  - `unresolved` — the target *plan or spec* is not in the corpus at all (likely out of scope for this run)
  Collapsing broken/unresolved loses debugging signal — never do that.
- Internal references that look internal but match a req_id in another plan are upgraded to `resolved` with a `[actually in <plan>]` annotation on `target_title`. The parser occasionally misclassifies these; the resolver corrects rather than reporting as broken.
- Standards refs carry `release_source` — `inline` (release came from the citation text itself) vs `doc_level` (filled from `RequirementTree.referenced_standards_releases`). Downstream consumers need to know which to score it correctly.
- One `CrossReferenceManifest` per source document. The corpus-wide manifest list preserves per-plan granularity.
- Resolver **never mutates** the input trees — manifests are a separate output.

**Key choices**
- Corpus-scoped index built once at construction — resolving N trees is `O(total_refs)` not `O(N × total_refs)`.
- Typed dataclasses per resolution flavor (instead of a single union) — each flavor has distinct fields (standards has spec/release; cross-plan carries a list of target IDs), and a union would erase type info at consumer sites.
- `ManifestSummary` is computed post-hoc from the ref lists, not tracked incrementally — easier to keep correct when a new `RefStatus` case is added.
- `broken` status is preserved in manifests (not filtered out) so downstream reports and the Web UI can render a "broken refs" list without re-walking trees.

**Non-goals**
- Not a standards fetcher — downloading specs, extracting sections, and building the standards graph live in [standards](../standards/MODULE.md).
- Not a semantic linker — concept-to-concept linking across plans ("these two requirements describe the same feature") is [taxonomy](../taxonomy/MODULE.md)'s job.
- No mutation of `RequirementTree`. Callers that need annotated trees join the manifest back in at query time or at graph-build time.
