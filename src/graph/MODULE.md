# graph

**Purpose**
Unified Knowledge Graph construction (TDD ¬ß5.8, D-002). Stitches the outputs of parser / resolver / taxonomy / standards into a single `networkx.DiGraph` that spans every MNO √ó release √ó doc_type in the corpus. The graph is the **routing layer** for queries ‚Äî it resolves scope ("VZW Feb2026 data-retry") to a set of requirement IDs; the vector store then ranks those.

**Public surface**
- Schema (schema.py):
  - `NodeType` enum: `MNO | Release | Plan | Requirement | Standard_Section | Feature`
  - `EdgeType` enum: `has_release | contains_plan | parent_of | belongs_to | depends_on | shared_standard | references_standard | parent_section | maps_to | feature_depends_on`
  - ID helpers: `mno_id`, `release_id`, `plan_id`, `req_id`, `std_section_id`, `std_spec_id`, `feature_id`
- Builder (builder.py):
  - `KnowledgeGraphBuilder` ‚Äî `.build(trees_dir, manifests_dir, taxonomy_path, standards_dir) -> nx.DiGraph`; exposes `.graph` and `.compute_stats()`
  - `GraphStats` ‚Äî `total_nodes`, `total_edges`, `nodes_by_type`, `edges_by_type`; `to_dict`, `save_json`
- CLI: `graph_cli.main`

**Invariants**
- **Single unified graph.** Scope (MNO, release, doc_type) is a node attribute / graph-traversal filter, not a separate graph instance. Changing this requires a D-002-level decision.
- Node IDs are deterministic and globally unique by construction ‚Äî the ID helpers (`mno_id`, `release_id`, etc.) are the only way to mint them. Any code that hand-builds IDs breaks idempotent rebuilds.
- Build is idempotent given the same inputs: starting from an empty `DiGraph`, the same tree/manifest/taxonomy/standards files produce the same nodes and edges every time.
- Build order matters ‚Äî requirement nodes exist before cross-ref edges link them; standards nodes exist before `references_standard` edges; feature nodes exist before `maps_to` edges. The `build()` sequence encodes this.
- `shared_standard` edges are derived, not declared ‚Äî two requirements are linked if they both reference the same standards section. This is why the step runs last.
- Graph does not embed text or carry LLM metadata ‚Äî that lives in [vectorstore](../vectorstore/MODULE.md). A node points back to its req_id / plan_id; downstream stages join back as needed.

**Key choices**
- `networkx.DiGraph` over a persistent graph database ‚Äî the whole corpus easily fits in memory; serialization to JSON/GraphML is cheap; no deployment footprint. Revisit if corpus size changes order of magnitude.
- Nodes typed by an attribute (`type: NodeType.*`) rather than by subclass ‚Äî keeps the graph homogeneous for NetworkX algorithms; callers filter with a predicate.
- ID prefixes (`mno:`, `release:`, `plan:`, `req:`, `std:`, `feature:`) make IDs self-describing in logs and compact reports (D-012) without consulting metadata.
- Standards graph has both spec-level (`std:24.301:11`) and section-level (`std:24.301:11:5.5.1.2.5`) nodes with `parent_section` edges between them ‚Äî query can answer "which requirements touch 24.301" via the spec-level node without enumerating sections.
- `shared_standard` edges are a denormalization: they could be computed from two hops, but the query path uses them often enough that the storage cost is worth it.

**Non-goals**
- Not a graph query engine ‚Äî NetworkX gives you traversal; LLM-facing query planning lives in [query](../query/MODULE.md).
- No ranking, no embedding, no similarity. A node is either there or it isn't; a path either exists or it doesn't.
- No persistence layer beyond the JSON sidecar artifacts the CLI writes ‚Äî the graph is rebuilt from source every pipeline run.
- Test_Plan / Test_Case node types from TDD ¬ß6 are deferred (Step 4) until test-plan docs are ingested; add the builder step when that data arrives.
