# corrections

**Purpose**
Per-environment profile + taxonomy correction handling: store engineer-edited overrides, diff them against pipeline output, and emit compact FIX reports (no proprietary content) that are pasteable back into chat. Implements the corrections-override pattern (D-011): the pipeline automatically prefers `<doc_root>/corrections/*.json` over its own output on the next run.

**Public surface** (enumerated in `__init__.py` via `__all__`)
- `FixReport` (schema.py) — compact diff payload: `env`, `artifact`, `lines`, `summary`; `to_text()`, `is_empty`
- `CorrectionStore` (store.py) — filesystem helper bound to an `EnvironmentConfig`:
  - path accessors: `profile_output_path()`, `profile_correction_path()`, `taxonomy_output_path()`, `taxonomy_correction_path()`
  - status: `profile_status()`, `taxonomy_status()`
  - load/save (typed): `load_profile_output/correction/effective`, `save_profile_correction`, same trio for taxonomy
  - lifecycle: `start_profile_correction` / `start_taxonomy_correction` (copy output → correction), `discard_*_correction`
  - raw JSON IO for form-backed editing: `read/write_profile_correction_raw`, `read/write_taxonomy_correction_raw`
- `profile_fix_report(output, correction, env_name)`, `taxonomy_fix_report(output, correction, env_name)` (compactor.py) — produce `FixReport`s

**Invariants**
- Corrections live at `<doc_root>/corrections/profile.json` and `<doc_root>/corrections/taxonomy.json`. Pipeline stages look here *first*; output is the fallback. This module owns those paths — nothing else should hardcode them.
- `load_*_effective()` returns correction if present, else output, else `None`. This three-state return is the contract — callers must handle `None` (no artifact at all).
- `FixReport` never contains proprietary document content — only field names, regex patterns, feature IDs/names, keyword tokens, and counts. Safe for chat paste. (Enforces D-012.)
- Raw JSON IO (`*_correction_raw`) is only for the Web UI form round-trip; typed IO is preferred everywhere else so schema drift surfaces as a load error, not silent data loss.

**Key choices**
- Correction files are a full copy of the effective artifact, not a diff — one file to read, one file to validate, and rollback is `discard_*_correction()`.
- `FixReport` is a plain dataclass with line-oriented text; formatting is deterministic so diffs across runs stay comparable in chat.
- `start_*_correction()` uses `shutil.copy2` — preserves mtime, which is useful when reviewing which run a correction was forked from.
- Explicit `__all__` in `__init__.py` — this is the canonical curated-surface example in the codebase (others get `__all__` as MODULE.md is curated).

**Non-goals**
- Not a schema validator — `DocumentProfile.load_json` / `FeatureTaxonomy.load_json` own validation. A malformed correction surfaces as a load error at stage start, not silent corruption.
- No git/history tracking — corrections are user-owned files; if versioning matters, the user commits them to their own repo.
- No UI rendering — Web UI templates render the dict from `read_*_correction_raw()`; this module only moves bytes and emits FIX text.
